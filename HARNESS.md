# HARNESS · 操作宪法（模型无关）

> 这是一套**以 markdown 文档为载体、模型无关的多 Agent 长篇小说创作 harness**。
> 它不是软件，没有可执行代码。它由三样东西构成：
> **① 角色卡**（`roles/`，可粘进任意大模型的系统提示）
> **② 接力协议**（`playbook/`，告诉你把什么贴给谁、输出存到哪）
> **③ 文档化的世界与记忆**（`world/ characters/ story/ memory/ …`，既是状态又是模型间传递的媒介）。
>
> 你可以给每个角色**指定不同的大模型**（ChatGPT / Claude / DeepSeek …，见 `cast.md`），
> 让它们各演各的，由你（或一个总控模型）居中接力，最终汇成长篇小说。

---

## 0. 运行方式：接力（文档为状态，CLI 为通道）

一切靠**接力**。接力有两种通道，**结构完全相同**，只是"谁把内容递到模型面前"不同：

- **方式 B · CLI 自动接力（推荐，默认）**：Showrunner（Claude Code 主会话）用 Bash 调度本机模型 CLI
  （`claude -p` / `codex exec` / `opencode run`）完成每一次接力，输出由 Showrunner 校验后落盘。
  协议见 **`playbook/0-cli-relay.md`**，选角与命令见 `cast.md`。
  允许的自动化**仅限"调用另一个模型 CLI 并落盘其输出"这一个动作**——仍没有业务脚本，文档仍是唯一状态源。
- **方式 A · 手动接力（降级通道）**：CLI 故障/限额，或角色被指派给只有网页的模型时，
  Showrunner 出"待粘贴块"，由用户贴到对应模型再贴回。内容与 B 的 prompt 完全一致，只是人肉递送。

> **v1 教训（ch-0001/0002）**：只有方式 A 时，粘贴摩擦导致所有角色都被总控 stand-in 代演，
> 多模型轮动名存实亡、maker-checker 失效。因此 v2 立规：**主会话永不扮演任何角色**（Claude 系角色
> 也走 `claude -p` 子进程），stand-in 需用户当场批准并计数（playbook/0 §4）。

```
        ┌──────────────────────────────────────────────┐
        │  SHOWRUNNER（你 或 一个总控模型会话）           │  ← 唯一上帝视角
        │  不创作，只做四件事：                            │
        │  组装上下文 · 接力转发 · 守闸门 · 落盘归档        │
        └───────────────┬──────────────────────────────┘
   按 cast.md 指定的模型，逐个开"干净的对话窗口"去扮演：
   ┌─────────┬─────────┬───────────┬─────────┬──────────┬──────────┐
   ▼         ▼         ▼           ▼         ▼          ▼
 架构师    导演    角色演员×N      总编    连续性     输出器
(任意模型)         (各角色可不同模型)
```

**一拍（beat）= 一次接力**，原子动作见 §4。子代理之间永不直接对话——
A 的发言由你写进场记，你再带着「场记 + B 的认知包」去问 B。

> 若你用 Claude Code 当 Showrunner（默认形态）：主会话就是上面的总控，它**只调度不创作**——
> 组 prompt、按 `cast.md` 调用对应 CLI、校验输出、落盘、守闸门（方式 B，见 `playbook/0-cli-relay.md`）；
> 仅当某条通道降级时才出"待粘贴块"走方式 A。
> `.claude/commands/` 里的 `/world-create`、`/scene-run` 等是这套流程的便捷启动器（可选）。

## 1. 三条公理（系统的灵魂）

### ① 信息隔离靠"构造"，不靠"自觉"
角色不能有上帝视角——**不是请求模型"假装不知道"，而是它的对话窗口里物理上就没有那条信息。**

在接力中这一点天然成立：你给某角色的是一个**干净的上下文**——方式 B 里是一次无状态 CLI 调用的 prompt，
方式 A 里是一个干净对话窗口——里面只装四样东西（见 §4）。
秘密、别人的内心、它不在场发生的事——**你根本不会装进去**，所以它演不出破绽。

铁律：
- **每个角色每次用各自独立、干净的上下文**（B：一次调度=一个干净上下文，且演员调用禁读任何文件、cwd 锁在空 run 目录；
  A：独立窗口，不在同一窗口串演两个角色）。
- 给角色的内容**只来自它的认知包 + 它亲历的公开场记**，绝不包含其他角色的认知包或 INNER。

### ② 文件系统是唯一事实源
对话会丢、会满、会串。**任何产物先落盘成 markdown，再继续。** 模型记不记得不重要，文件记得就行。

### ③ 闸门制
连续性检查 + 总编终审，任一不过 → 回退重做，**绝不带病发布**。每章发布前必须把状态变更落库到 `memory/`。

## 2. 角色清单与选角

七张角色卡在 `roles/`，外加一张总控卡 `roles/showrunner.md`。每张卡 = 一个可粘进任意模型的系统提示。

| 角色卡 | 身份 | 唯一可写 | 一句话职责 |
|---|---|---|---|
| `showrunner.md` | 总控/接力人 | （编排，不创作） | 组装上下文、接力、守闸门、落盘 |
| `world-architect.md` | 世界观架构师 | `world/**` | 立法者：定规则、驳回设定冲突 |
| `plot-director.md` | 剧情导演 | `story/**` | 设计弧线/章纲/场景 brief，下表演任务（不写台词） |
| `character-actor.md` | 角色演员（多开） | （只读，永不写档案） | 仅凭人设与认知演绎，禁上帝视角/禁 OOC |
| `editor.md` | 总编 | `reviews/**` | 终审，出 pass/revise/reject，可驳回 |
| `continuity-checker.md` | 连续性检查器 | `reviews/**` | 查前后衔接与物理连续性 |
| `prose-writer.md` | 小说输出器 | `drafts/**` | 把场记写成正文，不改剧情 |
| `memory-manager.md` | 记忆管理员 | `memory/**`, `context_packs/**` | 编认知包、出记忆补丁 |

**选角在 `cast.md`**：把每个角色映射到一个模型。例如让擅长中文文采的模型当 `prose-writer`、
让推理强的模型当 `editor`、给不同角色演员配不同模型制造"声音差异"。换模型只改 `cast.md`，不动角色卡。

## 3. 写权限矩阵（RBAC）

由你（Showrunner）人工把守。某角色的输出若试图改**不属于它**的文档 = 越权，丢弃并改派对应角色。

```
world/**            ← 仅 world-architect
characters/**       ← 仅 /character-create 流程（架构师+导演联署）；演员永不可写自己档案
story/**            ← 仅 plot-director
context_packs/**    ← 仅 memory-manager
memory/**           ← 仅 memory-manager（通过 memory_patch 应用）
transcripts/**      ← 仅 Showrunner 在演绎中追加
drafts/**           ← 仅 prose-writer
reviews/**          ← editor / continuity-checker（审报）；Showrunner 仅可落 rb-*（revision-brief）与分歧摘要
relay/**            ← 仅 Showrunner（调度留痕：每次 dispatch 的 prompt 与原始输出，playbook/0）
```

七条红线（与你的需求一一对应）：
1. 角色无上帝视角 → §1①。 2. 角色不改世界观 → 演员只读 `world/`，冲突上报架构师裁决。
3. 导演不逼 OOC → 导演只给目标/限制，台词出自演员；演员有权用 `FLAG` 拒演。
4. 输出器不改剧情 → 只做呈现层加工，新增情节即违规。 5. 总编可驳回。 6. 架构师可驳回设定冲突。
7. 每章必落库记忆补丁，否则不许开下一章。

## 4. 原子动作：怎么跑一"拍"（接力核心）

这是整套系统被使用得最频繁的操作。**方式 B 下**，下述"开窗口/粘贴"读作"组 prompt → 按 `cast.md` 调度
CLI → 校验 → 落盘"（`playbook/0-cli-relay.md` §1，四块内容一字不差地进 prompt）。以"让角色 B 演下一拍"为例：

1. **打开 B 的对话窗口**（用 `cast.md` 指派给 B 的模型；建议每个角色固定一个干净会话）。
2. **粘贴四块，仅四块**：
   - **【角色卡】**：`roles/character-actor.md` 全文（一次性，开窗口时贴）。
   - **【YOU ARE】**：`characters/<B>.md` 的人设要点（身份/欲望/恐惧/底线/voice/当前目标）。
   - **【你的私有认知包】**：`context_packs/<scene>.<B>.md` 全文（B 此刻知道/相信什么）。
   - **【公开场记 + 本拍指令】**：截至此刻**B 亲历的**公开场记（只含各拍 SPEAK/ACT，**不含任何 INNER**）
     ＋ 导演给本拍的指令（要 B 回应谁/什么情境，限制条件）。
3. **取回 B 的 7 字段输出**（SPEAK / ACT / INNER / BASIS / STATE_DELTA / KNOWLEDGE_GAINED / FLAG）。
4. **落盘**：把这一拍追加进 `transcripts/<scene>.md`，标注可见性（public / aside(谁听见) / solo）。
5. 轮到下一个角色时，**换它的窗口**，重复——并且**只把它该看到的公开场记**喂给它。

> 关键纪律：第 2 步里你**绝不**会粘进 B 不该知道的东西（别的角色的认知包、别人的 INNER、B 不在场的拍）。
> 这就是"信息隔离靠构造"在手动流程里的落地——不靠模型自律，靠你不递那张牌。

完整场景循环（定序、抽检、收场、SCENE RESULT）见 `playbook/4-scene-run.md`。

## 5. 章节生产流水线

```
建大纲 ─▶ 编认知包 ─▶ 场景演绎 ─▶ 场景验收 ─▶ 生成正文 ─▶ 机械核对 ─▶ 终审 ─▶ 记忆补丁+校验 ─▶ 发布
 导演      记忆管理     演员×N     检查器        输出器      检查器      总编      记忆管理         git
                          ▲          │                         │           │
                          └─重演该拍─┘                         └◀─ revision-brief 回退 ─┘
                          (≤2轮)                               (单闸≤2轮·全章≤4轮·耗尽→升级作者)
```

整章一键流程见 `playbook/chapter-pipeline.md`；各阶段也可单独按 `playbook/` 对应文件手动跑。
闸门：场景验收 / 机械核对 / 终审任一非 pass → 按 **§10 循环宪法**回退：落盘 revision-brief、计轮次、
预算耗尽升级作者。绝不带病前进，也绝不无限重做。

## 6. 上下文纪律（百万字不失控）

- **传路径，不传正文**：Showrunner 的总控对话里只流转「文件路径 + 一句话摘要」，不要把整章正文读回来。
- **三级滚动摘要**：scene → chapter（`story/chapters/*.md` 的"摘要"段）→ arc（`story/arcs/*.md` 的"摘要"段）。
  导演读"故事至今"时读摘要 + 开放线索，不读原文。
- **每个角色窗口只装自己那几 K**：人设 + 认知包 + 本场场记。**任何模型都不该装全书。**
- 总控对话快满了？把当前进度写进 `story/chapters/<id>.md` 的"流水线状态"段，关掉重开即可续跑。

## 7. ID 与命名约定

- 章节 `ch-0001`（4 位零填充）｜场景 `sc-0001-03`（章内序号）｜弧线 `arc-01`
- 角色用稳定 slug（文件名 = slug），如 `shen-yan`（主角）
- 事件 `evt-0001`｜伏笔 `fs-0001`｜线索 `thread-0001`
- 提交信息：`feat(ch-0001): <章标题>`；记忆更新随章节同一次提交

## 8. 多模型协作要点

- **换模型**：只改 `cast.md` 的映射；角色卡与文档不动。同一角色换模型不影响连续性，因为人设与记忆在文档里。
- **声音差异**：给男主/女主/反派指派不同模型，可天然产生语气差异；也可同模型靠 voice 段区分。
- **能力分工**：文采型模型→输出器；强推理→总编/架构师；性价比模型→连续性/记忆这类机械活。
- **一致性兜底**：模型再怎么换，世界规则以 `world/` 为准、记忆以 `memory/` 为准、是否过审以总编为准。

## 9. 端到端示例（简版，详见 README 与 `playbook/`）

> 目标：`ch-0001` / `sc-0001-01`——沈砚在碑亭镇遇到一名乔装的游方修士（实为外来细作），对方旁敲侧击打探无字碑林与开封之事。张力来自**信息不对称**：
> 读者与导演知道此人来路不正，沈砚不知道。

1. **导演**（其窗口）产出 `story/scenes/sc-0001-01.md`：目标、出场人、各自 `knowledge`、限制（细作不得暴露来路、沈砚不得识破）。
2. **记忆管理员**产出两份认知包：`context_packs/sc-0001-01.shen-yan.md`（只知"自幼在碑林拓碑、镇上传闻将开封"，不知对方来路）
   与 `…probe.md`（知自己是细作、任务是打探碑林之秘）。**两份内容不交叉。**
3. **演绎**：你在沈砚窗口、细作窗口之间接力（§4），逐拍写进 `transcripts/sc-0001-01.md`。
   转发时剥掉对方的 INNER——沈砚永远看不到细作"INNER: 套碑林的话"。
4. **输出器**把场记写成第三人称限知正文 `drafts/ch-0001.md`，读者能比沈砚"多看一层"，章尾留钩。
5. **连续性**+**总编**审核 → pass。
6. **记忆管理员**出补丁：新事件 `evt-0002` 的 `known_by` **只有细作**——于是**下一章**给沈砚编包时，
   他依旧不知情。信息隔离跨章自动延续。
7. 发布：`git commit -m "feat(ch-0001): 碑亭镇上的陌生人"`。

## 10. 循环宪法（Loop Engineering）

闸门只回答"过不过"；**循环宪法回答"不过之后怎么办"**。系统共五层循环，任何一层都必须五件齐全：
**入口**（什么触发）、**出口**（谁依据什么判据说 pass）、**预算**（最多几轮）、**反馈**（什么格式带什么回去）、**升级**（预算耗尽交给谁）。

| 层 | 循环 | 验证者 | 预算 | 出口 | 预算耗尽→升级 |
|---|---|---|---|---|---|
| L0 | 拍：指令→7字段→FLAG/抽检→重演该拍 | 总编抽检（每3–5拍） | 单拍重演≤2；beat_budget | 无 FLAG 且抽检过 | 回导演改本拍意图 |
| L1 | 场景：brief→演绎→**场景验收**→点名重演 | 连续性检查器 | 验收重演≤2轮 | 验收 pass | 回导演改 brief |
| L2 | 章：正文→机械核对→终审→回退重做 | 检查器＋总编 | 单闸≤2轮修订；全章累计≤4轮 | 双闸 pass | **作者裁决**（附分歧摘要） |
| L3 | 记忆：工作态→patch→应用→回读校验 | 记忆管理员+Showrunner | patch 修订≤2 | 校验 verified | Showrunner 人工比对 |
| L4 | 系统：弧线回顾→lessons→规则硬化 | 导演＋记忆管理员 | 每弧线 1 次 | 债务清零＋教训归档 | 作者 |

五条铁律（约束所有层）：
1. **反馈不落盘 = 没发生。** 任何重做必须有落盘的结构化修订指令：章/场级用 `templates/revision-brief.md` 成文件；
   拍级允许内联于场记（字段相同）。不允许口头回退。**演员窗口永远只收"作废声明＋新本拍指令"**，不收审稿证据原文。
2. **maker-checker 分离。** 自己不给自己发 pass；评审者只评不改（给证据与改法方向，不代写内容）；
   验证角色与生成角色**必须异模型**（三 CLI 在位时无借口，对照 `cast.md` maker-checker 矩阵；
   评审角色缺通道时宁可冻结，不得由 maker 同模型 stand-in——`playbook/0-cli-relay.md` §4 红线）。
3. **预算是一等公民。** 每轮回退计入章文件"流水线状态"；**预算耗尽不是失败，是升级**——
   整理双方立场与证据成分歧摘要，交作者裁决，拍板前冻结该章。
4. **重复问题必须硬化。** 审报中 `REPEAT` 标记的问题第 2 次出现 → 记入 `memory/lessons.md` 并硬化为规则
   （STYLE / 导演限制 / checklist / cognition_rule，流程见 `playbook/7-arc-retro.md`），不允许"再修一轮就好"。
   **lessons 永不进任何演员窗口**（防表演迎合）——信息隔离同样适用于元信息，教训只经硬化间接生效。
5. **便宜验证在前，昂贵验证在后。** 机械可判的事（追溯、物品、时间线、登记核对）归检查器；
   判断力（OOC、因果、伏笔策略、可读性）归总编；能在场景层拦住的，不要漏到章层。
