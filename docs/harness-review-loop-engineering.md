> 评审日期：2026-06-11。本文是元层文档，放 `docs/`（不进 `reviews/`——那里按 RBAC 只归总编/连续性写章节审报）。

## 0. 结论（TL;DR）

这套 harness 的三公理（信息隔离靠构造 / 文件是唯一事实源 / 闸门制）设计扎实，实跑 ch-0001 也验证了隔离链路确实守得住。**Loop Engineering 的原材料其实都已在场**：闸门、回退箭头、FLAG 拒演通道、beat_budget、INSUFFICIENT、幂等 patch——设计者显然有循环直觉。

问题在于：**系统把"循环"当异常路径，而不是一等公民。** 回退有箭头、无协议——

- 循环**无预算**：「回退重做」没说最多几轮，预算耗尽交给谁；
- 反馈**无契约**：`revise` 之后，重演的演员窗口里该贴什么？没有模板，全靠 Showrunner 现场发挥；
- 失败**无记忆**：审报第二轮直接覆盖第一轮，没人知道"这问题上轮就有"，更没有机制把教训硬化为规则。

改造方向**不是推翻，是把隐式循环显式化**：命名五层循环（L0 拍 / L1 场景 / L2 章 / L3 记忆 / L4 系统），每层补齐统一的「循环契约」五要素，新增两件套——`templates/revision-brief.md`（结构化反馈载体）与 `memory/lessons.md`（让循环改进循环的飞轮）。全部以纯文档协议落地，不引入任何代码，与项目宪法兼容。

## 1. 对照框架：Loop Engineering 是什么

业界近期（Osmani、MindStudio、Lushbinary 等）把多 agent 工作流的重心从 prompt engineering 转向 **loop engineering**："你不再是反复 prompt 的人，你是设计循环的人。" 一个工程化的循环必须五件齐全：

| 要素 | 含义 | 反面（坏味道） |
|---|---|---|
| **出口条件** | 谁、依据什么判据说"完成" | "改到满意为止" |
| **迭代预算** | 最多几轮，预算是一等设计项 | 无限重试 |
| **可信验证者** | 验证者与生成者分离（maker-checker），自己不给自己发 pass | 生成者自评通过 |
| **结构化反馈** | 失败时回流的不是原始 dump，而是定位+改法+保留范围+新旧错误区分 | "总编说不行，你重写一遍" |
| **升级路径** | 预算耗尽时升给谁裁决（通常是人） | 静默放水或无限烧 token |

与之配套的是 Anthropic evaluator-optimizer 模式的两条纪律：**评审者只评不改**；**验证尽量换模型**（本 harness 的 cast.md 天然支持）。下文按这五要素逐项体检。

## 2. 现状评估：已经做对的（保持，不要动）

1. **信息隔离链路完整且实跑有效**——`known_by` 过滤 → 认知包 → 可见性标签 → INNER 剥离 → `must_not_know` 只登记 id 不写内容。ch-0001 两场的"隔离自检"段显示链路真守住了（沈砚的起疑完全长在自己认知里）。这是全系统最强资产。
2. **FLAG 是现成的自下而上反馈通道**——演员可拒演 OOC，且 playbook/4 明确"角色是对的"。多数 multi-agent 系统缺这个。
3. **beat_budget 是全系统唯一显式循环预算**——证明直觉在，只是没推广到其它循环。
4. **回退格子已经存在**——总编的 `RETURN_TO: perform | prose | director | architect` 实际上是一张升级矩阵，只差预算与触发条件。
5. **上下文纪律**（传路径不传正文 / 三级摘要 / 断点续跑）正是循环系统说的"每轮上下文新鲜度"，方向完全正确。
6. **生成端自报口径已具雏形**——prose-writer 的 `beats_covered` 与 `INSUFFICIENT`、演员的 `BASIS`，都是"可验证生成"的接口（虽然目前没人去验，见 P6）。
7. **幂等 patch + 稳定 id**——为"重复应用"这种循环场景预留了正确语义。

## 3. 问题清单（按严重度排序，均给实跑证据）

### P1 · 循环无预算、无升级路径 〔严重〕
**现象**：HARNESS §1③/§5、playbook/6 只说「任一不过即回退」「绝不带病前进」——定义了方向，没定义**次数**。总编连续三轮 `revise` 怎么办？`reject` 后导演重设计又被 reject 怎么办？没有任何文档回答。
**风险**：两个失败模式——无限循环烧 token；或更糟：Showrunner 在第 N 轮"觉得差不多了"私下放水，闸门制名存实亡。
**对策**：每层循环定预算（建议：单闸门 ≤2 轮修订，单章全闸门累计 ≤4 轮），预算耗尽 = **强制升级给作者**，附分歧摘要。"卡住了去问人"本身就是合法出口，要写进协议。

### P2 · 反馈无契约 〔严重〕
**现象**：闸门产出 `BLOCKERS/MAJOR/MINOR` 后，playbook/6 只说「按 ISSUES 回退」。但回退到 perform 时，**重演的演员窗口里贴什么？** 哪些拍保留、哪些拍作废、带什么新限制、原认知包要不要改——没有模板，没有输入契约。回退到 prose 同理。
**风险**：循环最贵的一步（重做）质量完全取决于 Showrunner 的现场发挥；反馈在转译中失真，修了 A 漏了 B，下一轮闸门又挂——这正是循环不收敛的典型成因。
**对策**：新增 `templates/revision-brief.md`，作为一切回退的**唯一入口文档**。字段：attempt 编号 / 回退目标阶段 / 保留清单（明确不要动的）/ 修改清单（每条：证据→为什么→改法方向）/ 新增约束 / REPEAT 标记（上轮已出现）。总编与连续性的角色卡相应加一节"产出 revision-brief 所需素材"。

### P3 · 重复失败不可见，复审不增量 〔中〕
**现象**：审报固定写 `reviews/<ch>.editorial.md`，第二轮直接覆盖；没有 attempt 编号，没有"同一问题第二次出现"的标记；复审也没说可以只查上轮 BLOCKERS+受影响接缝，默认全量重审。
**风险**：测不出循环是否在**收敛**（同类问题反复出现 = 系统性病灶，应升级处理而不是再修一轮）；全量复审浪费最贵的总编上下文。
**对策**：审报命名加轮次 `reviews/<ch>.editorial.r2.md`；裁决格式加 `REPEAT:[...]` 字段；明确**增量复审**协议——复审输入 = 上轮报告 + revision-brief + 改动说明，优先核销旧 BLOCKERS。`REPEAT` 非空两次 → 直接触发 P7 的 lessons 流程。

### P4 · 场景级缺闸门，错误穿越两个昂贵阶段才被捕获 〔中·性价比最高的新增〕
**现象**：第一道正式闸门在**章级**（连续性，且在正文之后）。导演限制是否被守住（"柯九不得暴露修为"）、exit_condition 是否命中、该埋的伏笔是否真埋了（fs-0002）、隔离是否守住——这些**在场景演完当下就可机械判定**的事，要等全部场景演完+正文写完才被检查。
**证据**：ch-0001 的两份场记末尾都有 Showrunner 手写的「隔离自检」段——实践已经自发长出了这道检查，但它没有协议地位、没有独立验证者、没有 verdict 格式（违反 maker-checker：演绎的组装者自查自己）。
**对策**：playbook/4 增加**步骤 5 · 场景验收**：场记 vs brief 的四项机械核对（导演限制 / 出口条件 / 伏笔 plant·payoff / 隔离审计），指派给 continuity-checker（便宜模型，cast 现成）。verdict 落 `reviews/<sc>.scene-check.md`；不过 → 按 revision-brief 点名重演具体拍，预算 2 轮。SCENE RESULT 里的"隔离自检"保留为生成端自报，验收时核对。

### P5 · 章内状态传递靠默契（loop-carried state 未显式化）〔中〕
**现象**：`memory/` 在**章过审后**才落库——这是对的（防带病状态污染）。但 sc-0001-02 的认知包必须包含 sc-0001-01 的后果。实跑里它确实包含了（「你白日刚经历的」一节），可是**翻遍 memory-manager 角色卡与 playbook，没有任何一句要求"编后续场景的包要读本章已演场景的 SCENE RESULT"**。这次做对，纯因为同一个 Claude 会话顺手带了上下文。
**风险**：换模型、换会话、断点续跑后重编包——状态断裂，sc-02 的角色不记得 sc-01 发生过什么。这是典型的循环携带状态（loop-carried state）漏定义。
**对策**：memory-manager 职责一加一条明确公式：**编 sc-N 的包 = `memory/`（章前快照）+ 本章 sc-1..N-1 的 SCENE RESULT `proposed_deltas`（工作态）**。章文件加一段「本章工作态」滚动累积，过审后由正式 patch 取代。

### P6 · 验证者无分层：机械活占用最贵的判断力 〔中〕
**现象**：总编一次审 7 维，输入 = 草稿+场记+brief+world+characters+memory（playbook/6），上下文巨大。但其中"正文每个情节能否追溯到拍"（beats_covered 自报无人验证——**done 是声明不是证明**）、物品链条、时间线先后，是纯机械核对，不需要强推理模型。
**风险**：贵循环干便宜活，审得慢且容易看花眼；真正需要判断力的 OOC/因果/可读性反而被稀释。
**对策**：**便宜的循环跑在昂贵的循环之前**。闸门 A（连续性）扩为"机械核对包"：现有 7 项 + 正文-场记追溯抽查（拿 beats_covered 反向核对）+ inventory/时间线比对；总编瘦身为判断力 4 项（OOC / 因果与动机 / 伏笔策略 / 可读性）+ 战力裁决。两闸门顺序不变，输入各自裁剪。

### P7 · 系统级飞轮缺失：教训不沉淀 〔战略〕
**现象**：每轮审稿发现的问题，修完即丢。若输出器总爱加现代词、某演员总试图越认知边界，下一章还会再犯，仍靠总编人肉再抓。整个系统没有任何"把审稿教训硬化为生成约束"的机制。
**风险**：质量靠闸门兜底而不是源头改进，审稿成本恒定不降；百万字尺度下这是最大的复利损失。Loop Engineering 最有价值的一层恰恰是**让循环改进循环**。
**对策**：新增 `memory/lessons.md`（记忆管理员可写）：每次 `revise/reject` 落一条根因归类（哪个角色/哪类错/什么证据）。规则：**同类教训出现第 2 次 → 必须硬化**——变成 STYLE.md 新规则 / 导演限制模板新条目 / checklist 新项 / 角色卡 cognition_rule 补充，并在 lessons 里标记 `hardened→<去处>`。注意隔离纪律：**lessons 永不直接喂给演员**（防表演迎合），只硬化为导演限制与 STYLE 规则。

### P8 · 伏笔/线索债务无维护循环 〔低·但长篇致命〕
**现象**：`foreshadowing.md`/`threads.md` 只有 plant/open 进、payoff/close 出，没有定期盘点。总编每章问"该回收的回收了吗"，但"该"由谁、何时判定？百万字后 open 伏笔堆积，无人能答。
**对策**：新增**弧线回顾**（playbook/7-arc-retro.md）：每弧线终点，导演+记忆管理员盘点所有 open 伏笔/线索的 age 与拟回收点，逾期者显式裁决「延期（落新拟回收点）/ abandoned（注销）」，禁止无限期挂账。lessons 提炼也挂在这个节拍上。

### P9 · 零散问题 〔低〕

- **流水线状态无循环语义**：只有 `pending/done`，没有 attempt 计数与最近 verdict，断点续跑进不了"循环中段"（不知道这是第几轮、上轮为什么挂）。→ schema 扩展：`连续性: { r1: revise(3 issues), r2: pending }`。
- **git 从未初始化**：实测无仓库。README/playbook 设计了"每章一次提交、可回滚到任意世界状态"的检查点层——循环系统的回退依赖检查点，目前这层纯属纸面。→ 立即 `git init` 并补首笔提交。
- **run.sh 含 `--dangerously-skip-permissions` + 固定 session id**：有安全隐患且脆弱（session 失效即废）。→ 建议删除或降级为 README 一句话说明。
- **独场戏的环境刺激注入渠道未定义**：sc-0001-02 的"砚温、碑林回应"这次被塞进认知包「盲区」段（语义被借用——盲区本应只写"你不知道 X 存在"）。→ 明确：环境/世界事件由**导演本拍指令**注入（"世界拍"），认知包盲区恢复本义。

## 4. 改造蓝图：五层循环模型

把系统重画为五层嵌套循环，每层补齐契约五要素：

```
L4 系统循环   弧线回顾 → lessons 提炼 → 规则硬化（STYLE/限制/清单/角色卡）
              节拍:每弧线1次 ｜ 出口:债务清零+教训归档 ｜ 升级:作者
L3 记忆循环   SCENE RESULT 工作态 →(章过审)→ patch → 应用 → 回读校验
              预算:patch 修订≤2 ｜ 出口:校验通过 ｜ 升级:Showrunner 人工比对
L2 章循环     正文 → 闸A机械核对 → 闸B总编 →(revision-brief)→ 回退重做
              预算:单闸≤2轮·全章≤4轮 ｜ 出口:双闸 pass ｜ 升级:作者裁决
L1 场景循环   brief → 逐拍演绎 → 场景验收(四项机械核对) →(点名重演)
              预算:验收重演≤2轮 ｜ 出口:验收 pass ｜ 升级:回导演改 brief
L0 拍循环     本拍指令 → 7字段 → FLAG/抽检 →(重演该拍)
              预算:单拍重演≤2·beat_budget(已有) ｜ 出口:无FLAG+抽检过 ｜ 升级:回导演
```

**循环宪法**（拟新增为 HARNESS.md §10，约束所有层）：

1. 任何"重做"必须经由 revision-brief，**不允许口头回退**——反馈不落盘 = 没发生。
2. 验证者与生成者分离：自己不给自己发 pass；评审者只评不改（写改法方向，不代写内容）。
3. 预算是一等公民：每层循环的轮次记入流水线状态；**预算耗尽不是失败，是升级**——带分歧摘要交作者。
4. 同类问题第 2 次出现（REPEAT）→ 必须进 lessons 并硬化，不允许"再修一轮就好"。
5. 便宜验证在前，昂贵验证在后；机械可判的事不占用判断力。

## 6. 反模式：明确不建议做的

1. **不要写脚本实现自动循环**。项目宪法是"纯文档、无代码"，Loop Engineering 在这里的正确形态 = 循环写成协议、由 Showrunner（人或 Claude 总控）执行。Claude Code 当 Showrunner 时自然获得半自动化，无需破例。
2. **不要给演员加自评循环**。演员自审会诱发出戏与上帝视角；验证必须外置（这正是现有 FLAG 设计高明的地方——它只允许拒演，不允许自评通过）。
3. **不要把 lessons 喂给演员窗口**。系统教训对演员也是"不该知道的信息"——知道"系统在抓什么"会导致表演迎合。信息隔离原则同样适用于元信息，只硬化为导演限制与 STYLE 规则。
4. **不要追求一次全量重审**。增量复审 + 抽查旧项足够；全量重审既贵又把收敛信号淹没在噪音里。
